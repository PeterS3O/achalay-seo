Cómo se compara lo que Google conoce vs lo que el sitio conecta

Tres universos comparados en escala real. El sitemap declara todo lo que existe; GSC muestra lo que rankea; el crawl mide lo que se descubre siguiendo enlaces internos. La distancia entre el rojo y el verde es el problema estructural.

Lectura: Google conoce y muestra en resultados casi 100× más URLs de las que el sitio conecta internamente. El sitio depende del sitemap XML y de la memoria histórica de Google, no de una navegación propia. Cualquier cambio de algoritmo, pérdida de backlinks o reset del índice puede desaparecer ese tráfico sin forma de reconstruirlo.

La evidencia

La causa raíz: paginación por API JS, invisible para el crawler.

Al inspeccionar una categoría editorial, el HTML servido muestra solo 7 artículos. No hay enlaces <a href> hacia "página 2" — hay un botón que dispara fetch() a la API interna. La API sí conoce todo el catálogo, pero cada llamada devuelve JSON, no una URL indexable por Google.

¿Por qué esto es letal para SEO? Google puede renderizar JavaScript en su segundo pass, pero: (1) no siempre lo hace, (2) cuando lo hace, no asigna la misma autoridad que a URLs paginadas tradicionales, (3) las URLs cargadas por JS comparten la URL del padre (todas se ven como la misma página para el crawler), y (4) el sitemap XML es el único camino que Google tiene para descubrir el catálogo — si mañana lo perdiera, el sitio desaparece del índice progresivamente.
La prueba de que se puede resolver

Por qué /ebooks/ tiene 89% de cobertura crawl↔GSC

Mismo dominio, mismo equipo, mismo CMS — pero /ebooks/ tiene un ratio crawl:GSC del 89% mientras el blog editorial vive en 0,2%. La diferencia son tres decisiones de arquitectura.

La solución técnica

Cómo reparar el paginado para que Google lo indexe.

La API actual queda intacta — sigue sirviendo el JSON para la experiencia de usuario. Lo que se agrega es una capa de URLs HTML server-rendered que Google (y cualquier crawler) puede seguir. Es un patrón estándar en publicaciones editoriales grandes (Wired, TechCrunch, Bloomberg lo hacen así).

Paso 1

Crear URLs paginadas HTML propias

Cada página del listado debe tener su propia URL indexable. Formato recomendado:

/soluciones-y-tecnologia-alimentaria/ → página 1 (canonical de sí misma)
/soluciones-y-tecnologia-alimentaria/pagina/2/ → página 2
/soluciones-y-tecnologia-alimentaria/pagina/3/ → página 3
...
/soluciones-y-tecnologia-alimentaria/pagina/1426/ → última página

Alternativa aceptable (WordPress-style): /soluciones-y-tecnologia-alimentaria/page/2/. Cualquiera funciona — lo importante es que sea URL única, HTML, y accesible sin JS.

Paso 2

Renderizar el HTML en el servidor · no en el cliente

Cada URL debe devolver HTML completo con los 7 artículos ya renderizados, no un <div id="root"></div> vacío que se hidrata con JS. La API se sigue usando internamente, pero desde el servidor Node/Next.js:

// Next.js App Router · route /soluciones-y-tecnologia-alimentaria/pagina/[n]/page.tsx
export async function generateStaticParams() {
  // pre-generar todas las páginas del listado (1..1426)
  const res = await fetch("https://api.../notes/public/?category=...&page=1");
  const { totalPages } = await res.json();
  return Array.from({length: totalPages}, (_, i) => ({n: String(i + 1)}));
}
 
export default async function Page({params}) {
  const {n} = params;
  const data = await fetch(`.../notes/public/?category=...&page=${n}`).then(r => r.json());
  return <ArticleList notes={data.notes} currentPage={Number(n)} totalPages={data.totalPages} />
}

Con SSR (o SSG con revalidación de 60min), Google recibe HTML completo en el primer response. Sin dependencia de renderizado JS.

Paso 3

Enlaces prev/next reales con <a href>

Al pie del listado (o donde esté el actual "Cargar más"), reemplazar el botón JS por enlaces HTML reales:

<!-- reemplazar el botón "Cargar más" por: -->
<nav aria-label="Paginación">
  <a href="/soluciones-y-tecnologia-alimentaria/">« Primera</a>
  <a href="/soluciones-y-tecnologia-alimentaria/pagina/{n-1}/">‹ Anterior</a>
  <a href="/soluciones-y-tecnologia-alimentaria/pagina/{n+1}/">Siguiente ›</a>
  <a href="/soluciones-y-tecnologia-alimentaria/pagina/1426/">Última »</a>
</nav>

Nada de onClick={loadMore} ni router.push(). Enlaces <a href> puros — el usuario obtiene la misma experiencia (navegación entre páginas), Google obtiene rastreabilidad.

Paso 4

Meta tags correctas en cada página paginada

Cada página del listado tiene su propio <head>:

<!-- página 2 de la categoría -->
<link rel="canonical" href=".../soluciones.../pagina/2/" />
  → auto-referencial · NO apuntar a /soluciones-y.../ (bug clásico)
 
<title>Soluciones y tecnología alimentaria · Página 2 · The Food Tech</title>
<meta name="description" content="Artículos 8-14 sobre..." />

Notas: las etiquetas <link rel="prev|next"> las dejó de usar Google en 2019 — no son necesarias pero tampoco molestan. Lo crítico son (1) canonical propio por página, (2) title distinto por página incluyendo "· Página N", (3) meta description propia con snippet de los artículos de esa página.

Paso 5

Módulo de "artículos relacionados" en el template de nota

Complemento indispensable de la paginación. Al pie de cada artículo, un módulo con 4-6 links HTML a otros artículos:

<!-- al final del template de artículo -->
<section class="related-articles">
  <h2>También te puede interesar</h2>
  <a href="/soluciones-.../otra-nota-1/">...</a>
  <a href="/soluciones-.../otra-nota-2/">...</a>
  // 4-6 links en total, seleccionados por: misma categoría
  // + subcategorías compartidas + últimos publicados
</section>

Criterio de selección (en orden de prioridad): (1) misma subcategoría, (2) tags compartidos, (3) últimos publicados en la categoría. Todos como links HTML server-rendered — igual que ya hace /ebooks/.

P1 · #01

Paginación fake por API + ausencia de módulo de "artículos relacionados"

Este es el hallazgo técnico más importante de toda la auditoría. Explica por qué el 88,1% del tráfico orgánico llega a páginas que el crawler no puede alcanzar, por qué hay 15 categorías fantasma, y por qué ningún otro fix (metas, H1s, redirects, etc.) va a mover la aguja mientras esto no se resuelva primero. Los otros accionables mejoran páginas individuales — este arregla cómo Google descubre el catálogo entero.

Lo que está roto

1 · Paginación fake: las categorías editoriales (soluciones, packaging, métodos) muestran solo 7 artículos en el listado. Para cargar los siguientes hay un botón que dispara fetch() a la API interna. No hay enlaces HTML <a href="?page=2"> — el crawler no los ve, y Google (aunque a veces renderice JS) no les asigna la misma autoridad que a URLs paginadas reales.

2 · Sin módulo de relacionados: los artículos individuales no enlazan a otros artículos. Cada nota es un "punto terminal" — Google llega, lee, y no tiene camino para descubrir más.

Evidencia numérica del problema:
  • /soluciones-y-tecnologia-alimentaria/ · crawler alcanza 14, Google indexa 6.900 → ratio 1:493
  • /packaging-y-procesamiento/ · crawler 16, Google 4.154 → ratio 1:260
  • /metodos-de-control-y-regulaciones/ · crawler 9, Google 1.559 → ratio 1:173
  • 77% de las URLs indexables tienen 0-4 inlinks internos (casi huérfanas)
  • No existe capa intermedia de páginas con 10-24 inlinks → confirma ausencia de hubs y relacionados

La prueba de que se puede resolver: /ebooks/

La misma empresa, el mismo equipo, el mismo CMS — pero /ebooks/ tiene 89% de cobertura crawl:GSC (vs 0-0,6% del blog editorial). ¿Por qué? Porque hace 3 cosas que el blog no hace:

  1. Hub HTML consolidado en /ebooks/ con listado completo enlazado con <a href> reales (no paginación fake)
  2. Fichas individuales con módulo "Ebooks relacionados" al pie · 4-6 links HTML a otros ebooks de la misma temática
  3. Malla interna densa que permite al crawler navegar todo el catálogo desde cualquier ficha
El resultado medido: Google conoce 35 ebooks vía GSC. El crawler alcanza 81 URLs internas (más que Google) → cobertura sobre-100% del subcatálogo. Es el techo real de lo que se puede lograr con arquitectura editorial disciplinada.
